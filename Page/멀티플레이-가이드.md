# Impact Dash 2.0 — Colyseus 실시간 멀티플레이 가이드

## 1. 선택한 방식: "호스트 릴레이"

이 게임은 물리/AI/벽돌/아이템/전투 판정이 전부 `index.html` 안의 자바스크립트에 이미 완성되어 있습니다. 이걸 Node.js 서버로 통째로 옮겨 다시 짜는 건 시간이 매우 오래 걸리고 버그 위험도 큽니다.

그래서 가장 쉬운 방법으로 **"호스트 릴레이(host relay)"** 구조를 썼습니다.

- 방에 가장 먼저 들어온 사람 = **호스트**. 이 사람의 브라우저가 지금까지와 완전히 똑같이 게임을 계산합니다(원본 코드 그대로).
- 호스트는 초당 20회 "지금 다들 어디 있고, HP가 얼마고, 벽돌이 몇 개 남았는지" 스냅샷을 서버에 보내고, 서버는 이걸 나머지 참가자들에게 그대로 전달합니다.
- 호스트가 아닌 사람은 화면에서 직접 계산하지 않고, 받은 스냅샷을 그대로 그리기만 합니다. 대신 자기 방향키/마우스 입력을 서버 경유로 호스트에게 보내면, 호스트가 그 입력으로 그 사람의 캐릭터를 대신 움직여줍니다.

Colyseus 서버 자체는 게임 규칙을 전혀 모릅니다. 그냥 "누가 호스트인지 기억하고, 메시지를 이리저리 전달"만 하는 우체부 역할입니다. 그래서 서버 코드가 아주 짧고, 게임 쪽 코드는 거의 그대로 재사용됩니다.

**장점**: 가장 빠르게 붙일 수 있고, 기존 게임 로직을 다시 짤 필요가 없음.
**한계**: 호스트가 아닌 사람은 자기 핑(ping)만큼 입력이 화면에 반영되는 게 늦게 보일 수 있음(보간/예측 처리는 이번 1차 버전에는 없음). 호스트가 중간에 나가면 그 판은 끊깁니다(재시작 필요).

## 2. 받은 파일

| 파일 | 설명 |
|---|---|
| `index.html` | 원본 게임 + 온라인 멀티플레이 기능이 통합된 버전 |
| `colyseus.browser.js` | Colyseus 클라이언트 SDK(브라우저용 자체 번들). **`index.html`과 항상 같은 폴더에 있어야 합니다.** |
| `colyseus-server/` | 그대로 실행하면 되는 Colyseus 서버 프로젝트 |
| `colyseus-server/index.js` | 서버 진입점 |
| `colyseus-server/rooms/GameRoom.js` | 방/호스트/로비(준비·강퇴·정원)/메시지 릴레이 로직 (핵심) |
| `colyseus-server/test-lobby.js` | 로비 기능(호스트 지정, 준비 상태 전파, 강퇴, 정원 6명 제한)을 자동으로 검증하는 스크립트 — 삭제해도 무방 |

## 3. 서버 실행하기

```bash
cd colyseus-server
npm install
npm start
```

기본적으로 `ws://localhost:2567` 에서 대기합니다. 콘솔에 아래처럼 뜨면 정상입니다.

```
[Impact Dash] Colyseus 서버 실행 중: ws://localhost:2567
```

(제가 직접 서버를 띄우고 여러 클라이언트를 접속시켜 "먼저 들어온 사람이 호스트로 지정되는지", "입력/스냅샷이 올바른 방향으로만 전달되는지", "게스트 준비 상태가 전파되는지", "호스트만 강퇴할 수 있는지·강퇴가 실제로 되는지", "정원 6명을 넘기면 접속이 거부되는지"를 `test-lobby.js`로 전부 검증했고, 의도대로 동작했습니다.)

## 4. 게임에서 온라인으로 플레이하기

1. `index.html`을 로컬 웹서버로 띄웁니다(파일을 더블클릭해서 여는 `file://` 방식은 일부 브라우저에서 웹소켓/보안 정책 때문에 막힐 수 있습니다). 가장 간단한 방법:
   ```bash
   npx serve .
   ```
2. 시작 화면에서 **"🎮 싱글 플레이"**(기존 로컬 메뉴 그대로) 또는 **"🌐 멀티 플레이"**를 선택합니다.
3. 멀티 플레이를 고르면 **"🏠 방 만들기(호스트)"** / **"🔑 방 참가하기(게스트)"** 중 하나를 고릅니다.
   - 호스트는 방을 새로 만듭니다(서버 주소 + 방 코드 + 닉네임 입력).
   - 게스트는 같은 서버 주소 + 같은 방 코드 + 닉네임으로 접속합니다. 그 방 코드로 만들어진 방이 없거나 이미 6명이 꽉 찼으면 접속에 실패합니다.
4. 접속하면 **대기실(로비)** 화면으로 이동합니다.
   - 참가자 목록(이름/호스트 표시/준비 상태)이 실시간으로 보입니다.
   - **게스트**는 "준비하기" 버튼을 눌러 준비 완료 상태로 바꿔야 합니다.
   - **호스트**는 모드(개인전/팀전)·AI 인원수·플레이타임을 고를 수 있고, **게스트가 전원 준비 완료해야만** "게임 시작" 버튼이 활성화됩니다(게스트가 아무도 없으면 바로 시작 가능).
   - **호스트**는 각 게스트 옆의 "추방" 버튼으로 강제 퇴장시킬 수 있습니다.
   - 누구나 "🚪 방 나가기" 버튼으로 로비를 나갈 수 있습니다.
5. 방 하나는 **호스트 포함 최대 6명**까지 들어올 수 있습니다(서버에서 강제).
6. 호스트가 "게임 시작"을 누르면 모든 참가자의 화면에서 동시에 라운드가 시작됩니다.

## 5. 다른 네트워크(각자 다른 집)에서 같이 하려면

로컬(`localhost`)이나 같은 와이파이 안에서는 그대로 되지만, 서로 다른 인터넷망에 있는 친구와 하려면 서버를 인터넷에 공개된 곳에 올려야 합니다. 가장 쉬운 선택지:

- **Colyseus Cloud** (colyseus 제작사가 만든 배포 서비스, 설정이 제일 간단함) — https://cloud.colyseus.io
- Render.com, Railway.app, Fly.io 같은 일반 Node.js 호스팅에 `colyseus-server` 폴더를 그대로 올려도 됩니다.

배포 후에는 시작 화면의 "서버 주소" 칸에 `wss://내서버주소` (https면 wss, http면 ws)를 입력하면 됩니다.

## 6. 이번에 구현한 것 / 아직 다듬을 부분

**구현 완료**
- Colyseus 서버: 호스트 자동 지정, 입력/스냅샷 릴레이, 호스트가 나가면 다음 사람에게 호스트 자동 인계, **게스트 준비(ready) 상태 관리**, **호스트의 게스트 강퇴(kick)**, **방 정원 6명 제한**
- 시작 화면: **싱글 플레이 / 멀티 플레이 분리**, 멀티 플레이 안에서 **호스트(방 만들기) / 게스트(방 참가하기) 버튼 분리**
- 대기실(로비) 화면: 실시간 참가자 목록, 호스트/준비 상태 표시, 게스트 준비 버튼, 호스트 전용 강퇴 버튼, 방 나가기 버튼, 호스트 전용 게임 설정(모드/AI 인원수/플레이타임), 게스트 전원 준비 완료해야 시작 가능
- `test-lobby.js`로 호스트 지정/준비 상태 전파/강퇴 권한 체크/강퇴 동작/정원 6명 초과 시 접속 거부까지 직접 실행해 확인했습니다.

**추가로 넣은 게 좋겠다고 판단해 포함한 것**
- 호스트가 방을 나가면 서버가 자동으로 다음 사람을 새 호스트로 지정합니다(로비 목록에 즉시 반영됩니다).
- 연결이 끊기면(네트워크 문제 등) 알림을 띄우고 메뉴로 안전하게 돌아갑니다.
- 방 코드가 틀렸거나 방이 꽉 찼을 때 게스트에게 명확한 실패 메시지를 보여줍니다.
- 총원(호스트+게스트+AI)이 6명을 넘지 않도록 AI 인원수를 자동으로 줄여서 시작합니다.

**알려진 한계 (실제 두 기기로 테스트해보며 다듬으면 좋을 부분)**
- 비호스트 참가자는 보간(interpolation)이나 입력 예측이 없어서, 핑이 크면(같은 와이파이가 아니라 인터넷을 오가면) 다른 사람 움직임이 살짝 끊겨 보일 수 있습니다.
- 온라인 모드는 한 기기당 1명만 조작합니다(기존의 분할화면/멀티 컨트롤러 기능과는 같이 쓰지 않음).
- 호스트가 **라운드 진행 중**에 나가면 새 호스트로 지정되긴 하지만, 그 사람 화면은 벽돌/아이템을 직접 만든 적이 없어 게임이 자연스럽게 이어지진 않습니다(재시작 필요). 로비 단계에서의 호스트 이관은 정상 동작합니다.
- 상점/코인 지급은 각자 자기 기기에 저장된 지갑 기준으로 그대로 동작하도록 해뒀지만, 실제 여러 기기로 라운드를 끝까지 돌려서 순위/코인 지급까지 확인해보시길 권장합니다.
- 스냅샷 전송 빈도는 초당 20회로 해뒀습니다(`index.html`의 `NET_SNAPSHOT_INTERVAL` 값을 줄이면 더 자주 보내서 부드러워지지만 데이터량이 늘어납니다).
- 다음에 넣으면 좋을 것: 로비에 채팅/음성, 재접속(reconnect) 지원, 라운드 중 호스트 이관 완전 지원, 클라이언트 보간으로 비호스트 체감 부드럽게 하기.

## 7. 버그 수정 내역 (2차 수정)

플레이 중 발견된 두 가지 문제를 고쳤습니다.

**① 게스트 화면에 호스트(및 다른 사람)가 안 보이던 문제**
원인은 `renderOrder()` 함수가 화면에 그릴 캐릭터 목록을 `aiList`/`netPeerList`/`playerSlots`에서만 가져왔기 때문입니다. 그런데 호스트가 아닌 참가자(게스트) 쪽에서는 다른 사람들의 캐릭터가 이 세 배열이 아니라, 스냅샷을 받아 채워지는 `characters` 배열에만 존재했습니다(호스트만 `netPeerList`를 실제로 채워서 씀). 그래서 게스트 화면은 자기 자신만 그려지고 있었습니다. 이제는 실제로 "지금 그려야 할 전체 캐릭터" 기준인 `characters` 배열을 그대로 사용하도록 고쳤습니다(자기 캐릭터가 맨 위에 그려지는 순서는 그대로 유지).

**② 게임을 조금 하면 서버에서 튕기던 문제**
Colyseus는 일정 주기(핑)마다 클라이언트가 응답하는지 확인하고, 응답이 없으면 연결을 끊습니다. 기본 설정은 대략 3초 정도로 꽤 빡빡한데, 호스트 브라우저는 물리 연산·AI·렌더링·스냅샷 직렬화를 전부 혼자 처리하다 보니 프레임이 잠깐 무거워지면(예: 가비지 컬렉션, 캐릭터가 많아질 때) 그 안에 응답을 못 보내 끊길 수 있었습니다. 아래 세 가지로 완화했습니다.
- 서버의 핑 관용 시간을 늘림(`colyseus-server/index.js`, `pingInterval: 3000, pingMaxRetries: 5` → 약 15초까지 여유)
- 게스트가 입력을 보내는 빈도를 매 프레임(최대 초당 60~144회)에서 초당 30회로 줄여 불필요한 트래픽을 줄임
- 메인 루프(`loop()`)를 try/catch로 감싸서, 특정 프레임에서 예외가 나도 애니메이션 루프 자체는 멈추지 않고 다음 프레임에서 계속 진행하도록 함(예외로 화면이 완전히 멈춰버리는 상황 방지)

이 세 가지를 적용한 뒤 서버를 직접 띄우고 로비/강퇴/정원 테스트를 다시 돌려 정상 동작을 확인했습니다.

## 8. 실제로 튕기게 만든 진짜 원인 (3차 수정, 확정)

② 항목의 핑 타임아웃 완화는 예방 차원의 조치였고, 직접 보내주신 서버 로그(`Unexpected end of MessagePack data` 에러)를 보고 진짜 원인을 찾았습니다.

**원인**: 서버(`colyseus` 패키지)는 `0.15.x` 버전인데, 게임 화면(`index.html`)이 CDN에서 불러오던 클라이언트 라이브러리(`colyseus.js`)는 `0.16.x` 버전이었습니다. Colyseus는 서버와 클라이언트의 메시지 인코딩(msgpack) 방식이 버전마다 조금씩 달라서, 이 둘의 마이너 버전이 다르면 클라이언트가 보낸 메시지를 서버가 중간까지만 읽다가 깨진 데이터로 인식하고(`Unexpected end of MessagePack data`) 그 연결을 강제로 끊어버립니다. 로비에서 준비 완료 후 게임을 시작하면 게스트 쪽에서 초당 30회씩 `input` 메시지를 보내기 시작하는데, 바로 그 시점에 튕겼던 것과 정확히 들어맞습니다.

**확인 방법**: 실제로 0.16.x 클라이언트로 0.15.x 서버에 접속해 게임과 똑같은 모양의 메시지를 반복 전송하는 재현 스크립트를 만들어 돌려봤더니, 몇 번의 메시지만 지나도 서버 쪽에서 연결이 끊기는 걸 확인했습니다. 클라이언트를 서버와 같은 `0.15.x`(정확히는 `0.15.28`)로 맞추자 같은 테스트가 문제없이 통과했습니다.

**수정 내용**: `index.html`의 `<script src="https://unpkg.com/colyseus.js@...">` 부분을 `0.15.28`로 고정했습니다(전에는 `^0.16.0`이라 최신 0.16 버전이 불려오고 있었습니다). 앞으로 서버 쪽 `colyseus` 패키지 버전을 올리게 되면, 클라이언트 쪽 `colyseus.js` 버전도 같은 마이너 버전대(`0.X.y`의 `X`)로 맞춰줘야 합니다.

## 10. "Colyseus.Client is not a constructor" (4차 수정, 확정)

버전을 0.15.28로 맞추고 나니 이번엔 새로운 오류(`Colyseus.Client is not a constructor`)가 났습니다. 확인해보니 unpkg가 서빙하는 `colyseus.js@0.15.28`의 사전 빌드 파일(`dist/colyseus.js`) 자체에 문제가 있었습니다 — 브라우저에는 없는 Node.js 전용 기능(`https`, `http`, `net`, `tls` 등)을 제대로 빼지 못한 채로 번들링되어 있어서, 브라우저에서 그 파일을 불러오면 `Colyseus` 객체가 제대로 만들어지지 않았습니다(이 버전의 CDN 배포판 자체의 문제로 보입니다).

**수정 내용**: CDN에 더 이상 의존하지 않고, npm의 `colyseus.js@0.15.28` 패키지 소스를 직접 가져와 `esbuild`로 브라우저용으로 새로 번들링해서 **`colyseus.browser.js`** 파일로 이 프로젝트에 함께 포함시켰습니다. `index.html`은 이제 `<script src="colyseus.browser.js"></script>`로 이 파일을 직접 불러옵니다(인터넷의 CDN이 아니라 같은 폴더의 파일이므로, `index.html`을 옮길 때는 `colyseus.browser.js`도 항상 같이 옮겨야 합니다).

**검증**: 직접 만든 번들을 Node.js 환경에서 실행해 `Colyseus.Client`가 정상적인 생성자 함수인지 확인했고, 실제 서버에 접속해 방 생성 → 참가 → 서로를 인식 → ready 전송까지 웹소켓 통신이 정상적으로 이뤄지는 것까지 확인했습니다.

## 12. "Max payload size exceeded" 로 튕기던 문제 (5차 수정, 확정)

보내주신 서버 에러(`RangeError: Max payload size exceeded`, `ws/lib/receiver.js`)를 보고 원인을 정확히 찾았습니다.

**원인**: Colyseus 서버가 쓰는 `@colyseus/ws-transport` 패키지는 메시지 하나의 최대 크기(`maxPayload`)를 명시하지 않으면 **기본값이 겨우 4KB(4096바이트)**입니다. 그런데 이 게임의 `snapshot` 메시지(호스트가 초당 20회 보내는, 캐릭터 최대 6명 + 벽돌(라운드마다 보통 40~80개) + 아이템 + 폭탄 + 코인 정보를 통째로 담은 메시지)는 라운드가 시작되고 벽돌이 배치되자마자 거의 항상 4KB를 넘습니다. 그래서 게임을 시작하고 조금만 지나면(벽돌이 있는 스냅샷이 처음 전송되는 순간) 서버가 그 메시지를 "너무 크다"며 강제로 연결을 끊어버리고 있었습니다.

**확인 방법**: 실제 벽돌 80개 + 캐릭터 6명 정도로 만든 스냅샷의 크기를 재보니 약 5.3KB로 4096바이트를 넘었고, 수정 전 서버로 그 정도 크기의 메시지를 보내보니 정확히 같은 에러로 끊기는 걸 재현했습니다. `maxPayload`를 넉넉하게 늘린 뒤 같은 테스트를 돌리자 문제없이 전달됐습니다.

**수정 내용**: `colyseus-server/index.js`의 `WebSocketTransport` 옵션에 `maxPayload: 1024 * 1024`(1MB)를 추가했습니다.

## 13. AI 인원수를 0으로 골랐는데 1명이 들어와 있던 문제 (6차 수정)

**원인**: 온라인 로비의 "AI 인원수" 버튼이 싱글플레이 메뉴와 **같은 전역 변수**(`selectedAiCount`)를 공유하고 있었습니다. 그런데 싱글플레이 메뉴 쪽에는 "1인용(플레이어 1명)일 때는 AI 0명(=상대 없음)을 고를 수 없다"는 규칙이 있고, 온라인 모드는 항상 이 기기에서 1명만 조작하는 것으로 취급되기 때문에, 같은 변수를 공유하는 상황에서 이 규칙이 로비의 선택값을 되돌려버릴 여지가 있었습니다.

**수정 내용**: 온라인 로비가 더 이상 싱글플레이 메뉴와 변수를 공유하지 않도록, 로비 전용의 독립된 상태(`netLobbyMode`, `netLobbyAiCount`, `netLobbyRoundMinutes`)를 새로 만들었습니다. 로비의 모드/AI 인원수/플레이타임 버튼은 이제 이 변수들만 바꾸고, 실제로 "게임 시작"을 누르는 바로 그 순간에만 `netStartAsHost()`가 이 값들을 게임 로직에 그대로 반영합니다. 이제 로비에서 고른 값이 다른 코드에 의해 중간에 바뀔 여지가 없습니다.

## 15. 게스트 화면이 끊겨 보이던 문제 (보간 적용, 7차 수정)

**원인**: 호스트가 아닌 참가자(게스트) 화면은 물리 연산을 직접 하지 않고, 호스트가 초당 20회 보내주는 스냅샷을 받은 그 순간에만 캐릭터 위치를 그 값으로 딱 바꿔치기하고 있었습니다. 초당 60프레임으로 그리는 화면에서 위치가 초당 20번만(약 50ms마다) "뚝, 뚝, 뚝" 하고 순간이동하듯 갱신되니 뚝뚝 끊겨 보일 수밖에 없었습니다(호스트 자신은 60fps로 직접 계산하니 매끄럽게 보이는 것과 대조적).

**수정 내용**: 이제 스냅샷이 도착하면 위치를 바로 바꾸지 않고 "목표 위치"만 갱신해두고, 매 프레임(`netInterpolateEntities()`) 지금 위치에서 그 목표를 향해 부드럽게 이동(보간)시킵니다. 각도도 최단 방향으로 자연스럽게 회전하도록 처리했습니다. 나(게스트 자신의 캐릭터)를 포함해 화면에 보이는 모든 캐릭터에 동일하게 적용됩니다. 다만 부활처럼 "정말로 순간이동"해야 하는 경우(멀리 떨어진 곳으로 갑자기 이동하거나 죽음↔삶 상태가 바뀔 때)는 보간 없이 즉시 그 자리로 스냅되도록 예외 처리해서, 부활할 때 캐릭터가 맵을 가로질러 미끄러지듯 이동하는 어색함은 없게 했습니다. 체감 반응성과 부드러움 사이의 보간 속도(`NET_POS_SMOOTH`)는 20Hz 스냅샷 기준에 맞춰뒀고, 필요하면 `index.html`에서 이 값을 조절할 수 있습니다.

## 16. 게스트 쪽 폭탄 연출이 이상하던 문제 (8차 수정)

**원인**: 폭탄이 터질 때의 파티클/화면 흔들림/효과음 연출은 `explodeBomb()` 함수 안에서만 실행되는데, 이 함수는 물리 연산을 하는 **호스트에서만** 호출됩니다. 게스트는 스냅샷으로 "폭탄이 사라졌다"는 결과만 받을 뿐, "지금 막 터졌다"는 그 순간의 이벤트 자체를 받지 못해서 연출이 재생되지 않거나(그냥 스냅샷에서 조용히 사라짐), 어쩌다 한 번 걸린 스냅샷 프레임에만 살짝 보이는 식으로 부자연스러웠습니다.

**수정 내용**: 호스트가 `explodeBomb()`을 실행하는 바로 그 순간, `bombExplode` 이벤트(폭발 위치 포함)를 다른 참가자들에게 즉시 보내도록 했습니다. 게스트는 이 이벤트를 받으면 호스트와 똑같은 시각 효과(충격 파티클, 파편, 폭발음)를 그 자리에서 직접 재생합니다(벽돌 파괴/캐릭터 넉백 같은 실제 물리 결과는 이미 스냅샷으로 반영되고 있으므로, 여기서는 눈에 보이는 연출만 재현하면 충분합니다). 내 캐릭터가 폭발 범위 안에 있을 때만 화면 흔들림이 함께 재생되도록 해서, 호스트 쪽 연출 규칙과 동일하게 맞췄습니다.

## 17. 폭탄 범위가 검게 나오고 타이머/연출이 안 보이던 문제 (9차 수정)

**원인**: 실제 폭탄 객체는 `{ x, y, timer, maxTimer, ringAngle }` 필드를 쓰는데, 스냅샷으로 보낼 때는 실수로 `fuseTimer`라는(실제로는 존재하지 않는) 필드 이름으로 보내고 있었습니다. 그러다 보니 게스트 쪽 `bombs` 배열에는 `timer`/`maxTimer`/`ringAngle`이 전부 빠진 채로 도착했고, 이 값들로 색상 비율·타이머 텍스트·회전 애니메이션을 계산하는 렌더링 코드가 전부 `undefined`/`NaN`을 다루게 되면서 범위가 검게 칠해지고 타이머 숫자와 회전 연출이 나오지 않았습니다. 같은 이유로 아이템(`radius` 누락)과 코인(`radius`/`spin` 누락)도 조용히 깨져 있었을 가능성이 있습니다.

**수정 내용**: 필드를 하나하나 골라 보내다가 빠뜨리는 실수를 원천적으로 없애기 위해, 아이템/폭탄/코인은 이제 객체를 통째로(얕은 복사) 보내도록 `netSerializeState()`를 고쳤습니다. 앞으로 이 오브젝트들에 필드가 추가되더라도 자동으로 함께 동기화됩니다.

## 18. 왼쪽 위 상태 UI에 다른 참가자가 안 보이고, 내 닉네임이 적용 안 되던 문제 (10차 수정)

**원인 두 가지**를 함께 고쳤습니다.

- **상태 UI(HUD) 목록**: 좌상단 참가자 목록(`buildHud()`)은 게임이 막 시작되는 시점에 딱 한 번만 만들어졌습니다. 그런데 호스트 쪽은 그 시점에는 아직 다른 참가자(`netPeerList`)가 채워지기 전이었고, 게스트 쪽은 애초에 "나 혼자"인 상태로 목록을 한 번 만든 뒤 다시는 갱신하지 않았습니다. 그래서 다른 참가자 정보가 계속 빠져 있었습니다. 이제 호스트는 다른 참가자를 다 채운 뒤 목록을 다시 만들고, 게스트는 스냅샷을 통해 새로운 사람/AI가 처음 나타날 때마다 목록을 자동으로 다시 만들도록 고쳤습니다.
- **내 닉네임**: 로비 접속 화면에서 입력한 닉네임은 방 목록(로비)에는 정상적으로 반영되고 있었지만, 정작 실제 게임 캐릭터의 이름표는 호스트/게스트 모두 그냥 하드코딩된 "PLAYER"를 쓰고 있어서 반영되지 않았습니다. 접속할 때 입력한 이름을 저장해뒀다가, 실제로 게임이 시작되는 시점에 내 캐릭터 이름표에 그대로 적용하도록 고쳤습니다.

## 19. 무적 흰색 표시·스윙 애니메이션 보간, 아이템 원 안전장치, 접속 끊긴 유저 AI 자동 대행 (11차 수정)

**요청 세 가지**를 함께 처리했습니다.

- **무적일 때 흰색 깜빡임이 끊겨 보이던 문제**: 호스트는 매 프레임 `invincible`(무적 여부) 값을 계산하고 있었지만, 정작 게스트에게 보내는 스냅샷에는 이 값이 빠져 있었습니다. 그래서 게스트 화면에서는 무적 흰색 효과가 스냅샷이 올 때마다(초당 20회) 뚝뚝 나타났다 사라졌다 했습니다. `netSerializeState()`가 `invincible` 값을 보내도록 하고, `netApplySnapshot()`도 이 값을 받아서 적용하도록 고쳤습니다.
- **스윙 애니메이션이 끊겨 보이던 문제**: 차징(`chargeAmount`)과 스윙(`swingTimer`) 진행 상태는 초당 20회 스냅샷으로만 갱신되다 보니, 그 사이 프레임에서는 값이 멈춰 있어 뚝뚝 끊기는 것처럼 보였습니다. 이제 게스트 화면은 매 프레임(60fps) `netInterpolateEntities()` 안에서 호스트와 같은 규칙(`CHARGE_RATE`, `SWING_DURATION` 등)으로 값을 미리 조금씩 진행시켜 부드럽게 이어 보이도록 했고, 다음 스냅샷이 도착하면 실제 값으로 다시 맞춰집니다.
- **아이템 원이 게스트에서 제대로 안 보이는 문제**: 확인해보니 이전 수정(17차, 필드 전체 복사)으로 `radius` 값 자체는 이미 정상적으로 전달되고 있었습니다. 혹시 남아있을 수 있는 예외 상황에 대비해 아이템·코인을 그릴 때 `it.radius || 10`, `c.radius || 8` 같은 안전한 기본값을 추가로 넣어, 값이 비어 있어도 최소한 원이 아예 안 보이는 일은 없도록 방어 코드를 더했습니다.
- **접속이 끊기거나 방을 나간 유저를 AI가 자연스럽게 대신 조종**: 이전에는 게임 도중 누군가 나가면 그 캐릭터가 그대로 멈춰버렸습니다. 이제 호스트는 `peer-left` 메시지를 받으면(서버가 이미 보내주고 있던 메시지를 재활용) 해당 캐릭터를 네트워크 조종 목록(`netPeerList`)에서 빼고 AI 조종 목록(`aiList`)에 넣어 즉시 AI가 이어받도록 했습니다. 이때 캐릭터의 `netSessionId`는 그대로 유지해서, 다른 참가자들 화면에서는 그 캐릭터가 갑자기 사라지거나 다시 나타나는 것처럼 보이지 않고 자연스럽게 이어집니다. 이름표에는 `(AI 대행)`이 자동으로 붙습니다.
  - **한계(그대로 유지)**: 라운드 도중 **호스트 본인**이 나가는 경우는 이번 수정 대상이 아닙니다. 호스트는 벽돌/아이템 배치를 포함한 게임 전체 상태를 혼자 들고 있기 때문에, 다른 참가자가 새 호스트로 지정되어도 그 상태를 이어받을 방법이 없어 그 판은 그대로 종료되고 새로 시작해야 합니다.

## 20. 게스트가 코인을 못 줍던 문제 + 폭탄 범위 애니메이션이 느려 보이던 문제 (12차 수정)

**두 가지 문제**를 함께 처리했습니다.

- **게스트가 코인을 못 줍는 문제**: 코인 줍기 판정 함수(`updateCoinPickup`)가 `isHuman(ch)`(=화면 왼쪽 하단 분할 패널에 있는 "내 캐릭터"인지)만 확인하고 있었습니다. 그런데 멀티플레이 호스트 쪽 시뮬레이션에서, 게스트가 조종하는 캐릭터(`netPeerList`)는 호스트 화면 기준으로는 "내 캐릭터"가 아니기 때문에 `isHuman()`이 항상 false였고, 그래서 게스트가 코인 위를 아무리 지나가도 전혀 줍히지 않았습니다. AI로 넘어간 캐릭터만 제외하는 `ch.isAI` 기준으로 바꿔서, 게스트가 조종하는 캐릭터도 코인을 주울 수 있게 했습니다.
  - **추가로 발견한 문제**: 코인을 주우면 늘어나는 "지갑(코인)"은 사실 각자 브라우저(기기)마다 따로 저장되는 개인 지갑입니다. 그런데 게스트가 주운 코인을 그냥 호스트 쪽에서 지갑에 더해버리면, 실제로는 게스트가 아니라 **호스트의** 지갑이 늘어나는 잘못된 결과가 됩니다. 그래서 게스트가 주운 코인은 호스트가 직접 지갑에 넣지 않고, "이 세션이 코인을 주웠다"는 알림을 그 게스트에게만 보내서(`event` 메시지, `type: 'coinPickup'`) 게스트 자신의 브라우저에서 지갑이 늘어나도록 고쳤습니다. 호스트 자신의 캐릭터는 예전처럼 즉시 지갑에 반영됩니다.
- **게스트 화면에서 폭탄 범위 링/타이머 애니메이션이 느리게(뚝뚝 끊겨) 보이는 문제**: 폭탄(`bombs`) 배열은 스냅샷이 올 때마다(초당 20회) 통째로 교체될 뿐, 그 사이 프레임에는 전혀 갱신되지 않았습니다. 그래서 폭발 범위를 표시하는 점선 링 회전과 카운트다운이 60fps가 아니라 20fps 수준의 계단식으로 보였습니다(특히 링 회전이 눈에 띄게 어색했습니다). 캐릭터 움직임에 보간을 적용했던 것과 같은 방식으로, `netInterpolateEntities()` 안에서 호스트의 `updateBombs()`와 동일한 공식으로 매 프레임 로컬에서 타이머/링 회전을 미리 조금씩 진행시키도록 했습니다. 다음 스냅샷이 도착하면 실제 값으로 다시 맞춰지므로 어긋날 걱정은 없습니다.

## 21. 코드에서 어디를 보면 되는지

`index.html`에서 `Ctrl+F`로 `온라인 멀티플레이` 또는 `netConnectAsHost`, `netConnectAsGuest`, `netRenderLobby`, `netToggleReady`, `netKickPeer`, `netStartAsHost`, `netStartAsClient`, `netSerializeState`, `netApplySnapshot`, `netInterpolateEntities`, `updateOnlineClient` 를 검색하면 관련 코드가 전부 한 블록에 모여 있습니다(주석도 한글로 자세히 달아뒀습니다). AI 자동 대행 로직은 `netBindRoomHandlers()` 안의 `peer-left` 메시지 처리 부분을, 코인 지갑 귀속 로직은 `updateCoinPickup()`과 `netHandleEvent()`의 `coinPickup` 부분을 검색하면 됩니다.

`colyseus-server/rooms/GameRoom.js`에는 `ready`/`kick` 메시지 처리와 `ROOM_MAX_CLIENTS`(정원) 로직이 있습니다.

## 22. 로컬(같은 컴퓨터)이 아니라 인터넷으로, 멀리 있는 친구와 플레이하기

지금까지 테스트는 같은 컴퓨터에서 브라우저 탭 두 개로 `ws://localhost:2567`에 접속하는 방식이었습니다. 인터넷 너머 다른 컴퓨터에 있는 친구와 하려면, Colyseus 서버(`colyseus-server`)를 **인터넷에서 접속 가능한 주소**로 열어줘야 합니다. 아래 두 가지 방법 중 하나를 쓰면 됩니다.

### 방법 A. Railway에 서버 올리기 (추천, 영구적인 주소)

1. https://railway.app 에서 무료로 회원가입합니다(신용카드 없이도 가입 가능, 매달 일정 무료 크레딧 제공).
2. 컴퓨터에 Railway CLI를 설치합니다: `npm install -g @railway/cli`
3. 터미널에서 서버 폴더로 이동: `cd colyseus-server`
4. 로그인: `railway login` (브라우저가 열리며 로그인 진행)
5. 새 프로젝트 만들기: `railway init`
6. 배포: `railway up` — 지금 폴더(`colyseus-server`)를 그대로 업로드해서 실행해줍니다. `index.js`가 이미 `process.env.PORT`를 쓰도록 되어 있어서 별도 코드 수정 없이 바로 됩니다.
7. 배포가 끝나면 Railway 대시보드에서 이 프로젝트를 열고 **Settings → Networking → Generate Domain**을 눌러 공개 도메인을 하나 받습니다. `무언가.up.railway.app` 같은 형태로 나옵니다.
8. 이 도메인 앞에 `wss://`를 붙인 것이 서버 주소입니다. 예: `wss://무언가.up.railway.app`
9. 게임 화면의 "서버 주소" 입력칸에 위 주소를 넣고, 호스트는 방을 만들고, 친구는 게스트로 같은 주소 + 방 코드를 입력해서 접속하면 됩니다.

한 번 배포해두면 컴퓨터를 꺼도 서버는 계속 켜져 있으므로, 매번 새로 설정할 필요 없이 방 코드만 바꿔가며 계속 쓸 수 있습니다.

### 방법 B. ngrok로 임시 터널 열기 (빠르지만 임시용)

당장 5분 안에 딱 한 판만 같이 하고 싶을 때 쓰는 방법입니다.

1. https://ngrok.com 에서 무료 가입 후 ngrok을 설치합니다.
2. 평소처럼 내 컴퓨터에서 서버를 켭니다: `cd colyseus-server && npm start`
3. 새 터미널 창에서: `ngrok http 2567`
4. 화면에 나오는 `https://무언가.ngrok-free.app` 형태의 주소를 복사해서, `https://`를 `wss://`로 바꿔 사용합니다. 예: `wss://무언가.ngrok-free.app`
5. 이 주소를 "서버 주소"에 입력해서 호스트/게스트가 접속합니다.

**주의**: 2026년 기준 ngrok 무료 플랜은 한 세션이 2시간 정도로 제한되고, 재시작할 때마다 주소가 매번 바뀝니다. 오래 즐기려면 중간에 ngrok을 다시 켜고 새 주소를 서로 다시 공유해야 하는 번거로움이 있습니다. 계속 같은 주소로 편하게 쓰고 싶다면 방법 A(Railway)를 추천합니다.

### 친구가 준비해야 할 것

서버 접속 주소만 바뀔 뿐, 게임 자체(그래픽/물리 계산)는 각자의 컴퓨터에서 돌아갑니다. 그래서 친구도 반드시 아래 두 파일을 **같은 버전으로** 받아서 자기 컴퓨터에서 열어야 합니다(카카오톡, 구글 드라이브 등으로 전달).

- `index.html`
- `colyseus.browser.js` (반드시 `index.html`과 같은 폴더에 있어야 함)

## 23. 전체 코드 점검 (13차 수정) — 게스트 코인/폭탄 애니메이션, 모바일 벽돌 컬링, 호스트 이탈 시 방치 문제

요청대로 싱글/멀티플레이 게임 로직 전체(`index.html`)와 서버(`colyseus-server`)를 다시 훑어보며 문제가 있어 보이는 부분을 고쳤습니다.

- **게스트가 코인을 못 줍던 문제**: 코인 줍기 판정이 "화면 왼쪽 하단의 내 캐릭터인지"만 확인하고 있어서, 호스트 화면 기준으로는 게스트가 조종하는 캐릭터가 그 조건에 걸리지 않아 아예 판정되지 않았습니다. AI가 아닌 캐릭터는 모두 주울 수 있도록 기준을 바꿨습니다. 추가로, 지갑(코인)은 브라우저(기기)마다 따로 저장되는데 게스트가 주운 코인을 그대로 처리하면 호스트의 지갑이 늘어나는 문제가 있어, 실제로 주운 사람에게만 반영되도록 고쳤습니다.
- **게스트 화면 폭탄 범위 애니메이션이 느리게 보이던 문제**: 폭탄 상태(카운트다운, 점선 링 회전)가 초당 20회 스냅샷으로만 갱신되고 그 사이에는 멈춰 있어서 뚝뚝 끊겨 보였습니다. 캐릭터 움직임처럼 매 프레임 로컬에서 미리 진행시키는 방식으로 부드럽게 만들었습니다.
- **모바일에서 벽돌이 가까이 가야만 보이던 문제 (요청하신 부분)**: 원인은 화면 밖 벽돌을 안 그리는 최적화(컬링) 로직이 항상 "줌 1배" 기준의 좁은 화면 범위로만 판정하고 있었던 것이었습니다. 그런데 모바일에서는 시야를 넓히기 위해 화면을 축소(camZoom<1)해서 그리기 때문에, 실제로는 더 넓은 범위가 보이는데도 컬링 판정은 여전히 좁은 범위만 "화면 안"이라고 여겨서, 카메라(캐릭터)가 벽돌에 가까이 가야만 그 좁은 판정 범위 안에 들어와 비로소 그려지는 문제였습니다. 컬링 판정 범위도 camZoom만큼 함께 넓히도록 고쳤습니다(데스크톱처럼 camZoom=1일 때는 기존과 동일하게 동작합니다).
- **(새로 발견) 라운드 도중 호스트가 나가면 게임이 조용히 망가지던 문제**: 이전까지는 "호스트가 도중에 나가면 그 판은 끊긴다"고만 문서에 적어뒀는데, 실제로 코드를 다시 보니 그보다 더 안 좋은 상황이 벌어질 수 있었습니다. 호스트가 나가면 Colyseus 서버가 남은 사람 중 아무나를 새 "신호 호스트"로 지정하는데, 이 게임은 벽돌/아이템/다른 사람 캐릭터 정보를 전부 원래 호스트 한 명의 브라우저 메모리에만 들고 있는 구조라서, 새로 지정된 사람은 그 정보를 하나도 갖고 있지 않습니다. 그런데도 예전 코드는 이걸 그대로 따라가서 그 사람이 게임 시뮬레이션을 돌리기 시작해버려, 다른 사람들 화면이 멈추거나 깨진 상태로 계속 진행되는 것처럼 보이는 문제가 있었습니다. 이제는 참가자들이 각자 이 상황(라운드 시작 때의 호스트와 지금 서버가 알려주는 호스트가 다름)을 감지하면, 조용히 망가진 채로 계속하는 대신 안내 메시지를 띄우고 방을 나가 대기 화면으로 돌아가도록 고쳤습니다. 새로 방을 만들어 다시 시작하면 됩니다.

### 코드에서 어디를 보면 되는지 (추가)

- 코인 지갑 귀속: `updateCoinPickup()`, `netHandleEvent()`의 `coinPickup` 처리
- 폭탄 보간: `netInterpolateEntities()` 안의 `bombs.forEach` 부분
- 모바일 벽돌 컬링: `renderPane()` 안의 `brickCullLeft`/`brickCullRight`/`brickCullTop`/`brickCullBottom`
- 호스트 이탈 감지/정리: `netRoundHostSessionId`, `netHostMigrationHandled`, `netHandleHostLostMidRound()` (room-info 메시지 핸들러 안에서 호출됨)

친구는 이 `index.html`을 더블클릭해서 브라우저로 열고, "멀티모드 → 게스트로 참가" → 서버 주소(위에서 만든 `wss://...` 주소)와 방 코드를 입력하면 됩니다.
